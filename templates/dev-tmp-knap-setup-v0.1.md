# Filename: setup-[shorthand].md

/* Setup lifecycle file. This is a first-class shard file at the same tier as init-<sh>.md.
   It contains human/agent-readable setup instructions ŌĆö everything needed to prepare
   the shard for use on a new machine or workspace.
   
   Required when shard.yaml declares `setup: full|flint|local`.
   
   In dev folders: `dev-setup-<sh>.md` (prefix stripped on install to `setup-<sh>.md`).
   
   This file is NOT a type, NOT in install/, NOT in a subfolder.
   It lives at the shard root alongside the init file. */

```markdown
# Setup [Shard Name]

One-time setup actions for [Shard Name].

/* Describe what needs to happen before this shard is usable.
   Be specific and actionable ŌĆö agents will follow these steps autonomously. */

# Prerequisites

/* List any tools, accounts, or access needed before starting setup */

- [Tool or access requirement]
- (continue)

# Actions

/* Step-by-step setup instructions. Each step should be independently verifiable. */

1. [First setup action ŌĆö be specific about commands, paths, or configuration]
2. (continue)

# Verification

/* How to confirm setup is complete */

- [Verification step ŌĆö what to check]
- (continue)

/* After all actions and verification pass, update the shard state file:
   
   For flint scope: Shards/(Shards) State/(Shard) [Name].md
   For local scope: Shards/(Shards) Local State/(Shard) [Name] (Local).md
   
   Set the frontmatter `setup` field to `completed`:
   
   setup: completed */
```

## State File Structure

The installer creates state files automatically when `setup:` is declared. Authors do not hand-write these, but must know the shape because setup verification reads/writes the `setup` field.

```markdown
---
id: [uuid4, generated by installer]
setup: required
---

# [Shard Name] ŌĆö Shard State

/* Optional: authors and agents may record setup progress, machine details, or
   credentials references here. The only field the runtime reads is `setup:`. */
```

| Field | Values | Meaning |
|-------|--------|---------|
| `setup` | `required` | Setup has not been completed ŌĆö `flint shard start` shows the SETUP REQUIRED banner |
| `setup` | `completed` | Setup is done ŌĆö banner suppressed |
| `setup` | `not-required` | This machine/repo does not need this scope of setup (e.g., `flint` scope satisfied but `local` does not apply) ŌĆö banner suppressed |

Locations:

| Scope | File(s) |
|-------|---------|
| `full` | `Shards/(Shards) State/(Shard) <Name>.md` (tracked) AND `Shards/(Shards) Local State/(Shard) <Name> (Local).md` (gitignored) |
| `flint` | `Shards/(Shards) State/(Shard) <Name>.md` (tracked) only |
| `local` | `Shards/(Shards) Local State/(Shard) <Name> (Local).md` (gitignored) only |

## Design Guidelines

- Write for agents AND humans ŌĆö both should be able to follow the instructions
- Be explicit about paths, commands, and expected outcomes
- Include verification steps so the setup can be confirmed programmatically
- Don't use Node.js scripts ŌĆö the setup file replaces `requires.workspace` with prose
- Keep it focused on what needs to happen, not why (the init file covers the why)
- If setup differs by machine (credentials, local paths), note which steps are per-machine
