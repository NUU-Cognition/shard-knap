# Naming and Placement

| | |
|---|---|
| Dev source path | `dev-setup-<sh>.md` (at the shard root, alongside `dev-init-<sh>.md`) |
| Installed path | `setup-<sh>.md` (the installer strips `dev-`) |
| Wikilink form | `[[setup-<sh>]]` (canonical) |

**Required when** `shard.yaml` declares `setup: full | flint | local`. The installer **refuses to install** a shard that declares `setup:` without this file. Shards with no `setup:` field do not need it.

**Not a type. Not in `install/`. Not in a subfolder.** This file is a first-class shard file at the same tier as the init file. Setup instructions are human/agent-readable — everything needed to prepare the shard for use on a new machine or workspace.

```markdown
# Setup [Shard Name]

One-time setup actions for [Shard Name].

/* Describe what needs to happen before this shard is usable.
   Be specific and actionable — agents will follow these steps autonomously. */

# Prerequisites

/* List any tools, accounts, or access needed before starting setup */

- [Tool or access requirement]
- (continue)

# Actions

/* Step-by-step setup instructions. Each step should be independently verifiable. */

1. [First setup action — be specific about commands, paths, or configuration]
2. (continue)

# Verification

/* How to confirm setup is complete */

- [Verification step — what to check]
- (continue)

/* After all actions and verification pass, mark setup complete:
   
     flint shard setup <name> --complete
   
   The default scope is derived from `manifest.setup`: `'full'` → both layers,
   `'flint'` → committed layer only, `'local'` → local layer only. Pass
   `--scope flint|local|both` to override. To reset, pass `--reset`.
   See [[knw-knap-manifest]] § `setup` for the full command surface. */
```

## State File Structure

The installer creates state files automatically when `setup:` is declared. Authors do not hand-write these, but must know the shape because setup verification reads/writes the `setup` field.

```markdown
---
id: [uuid4, generated by installer]
setup: required
---

# [Shard Name] — Shard State

/* Optional: authors and agents may record setup progress, machine details, or
   credentials references here. The only field the runtime reads is `setup:`. */
```

| Field | Values | Meaning |
|-------|--------|---------|
| `setup` | `required` | Setup has not been completed — `flint shard start` shows the SETUP REQUIRED banner |
| `setup` | `completed` | Setup is done — banner suppressed |
| `setup` | `not-required` | This machine/repo does not need this scope of setup (e.g., `flint` scope satisfied but `local` does not apply) — banner suppressed |

Locations:

| Scope | File(s) |
|-------|---------|
| `full` | `Shards/(Shards) State/(Shard) <Name>.md` (tracked) AND `Shards/(Shards) Local State/(Shard) <Name> (Local).md` (gitignored) |
| `flint` | `Shards/(Shards) State/(Shard) <Name>.md` (tracked) only |
| `local` | `Shards/(Shards) Local State/(Shard) <Name> (Local).md` (gitignored) only |

## Design Guidelines

- Write for agents AND humans — both should be able to follow the instructions
- Be explicit about paths, commands, and expected outcomes
- Include verification steps so the setup can be confirmed programmatically
- Don't use Node.js scripts — the setup file replaces `requires.workspace` with prose
- Keep it focused on what needs to happen, not why (the init file covers the why)
- If setup differs by machine (credentials, local paths), note which steps are per-machine
